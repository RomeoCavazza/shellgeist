# ShellGeist — Spécification technique (audit critique)

Document généré pour décrire variables, fonctions et rôles de chaque fichier.
Architecture : démon Python (socket Unix) + client Neovim (Lua) ; protocole JSON-lines.

---

## 1. Vue d’ensemble

- **Entrée** : `shellgeist` (CLI) ou plugin Neovim → envoi de but (goal) + mode (auto/review).
- **Backend** : un processus par workspace ; boucle agent (LLM + outils) ; événements streamés vers le client.
- **Outils** : list_files, read_file, write_file, find_files, run_shell, edit_file, sessions shell, etc.
- **Politique** : LoopGuard (répétitions, blocages), RetryEngine (transient), commandes dangereuses bloquées.

---

### 2.9 shellgeist/runtime/policy.py

- **record_outcome(tool_name, args, result)** → (blocked: bool, msg)

**RetryConfig** (dataclass) : max_attempts, max_total_retries, base_backoff_ms, max_backoff_ms ; from_env()

- __init__(config)
- **run_async(key, operation, classify_result, on_retry)** : boucle retry sur résultat "transient"

**_BLOCKED_PATTERNS** : list[re] (rm -rf /, mkfs, dd, mv vers /, >/dev/)

- **is_blocked(command)** → bool
- **classify_result_payload(res, tool_name)** → (transient|None, reason) (timeout, rate limit)

## 5. Flux de données (résumé)

1. **Neovim** : :SGAgent goal → init.lua run_agent(goal) → rpc.request(socket, {cmd="agent_task", goal, root, mode}) → lecture lignes JSON (execution_event | result).
2. **Backend** : server.client_handler reçoit agent_task → handle_request → Agent.run_task(goal, writer, session_id, mode, reader) → build_system_prompt, load_recent_history, append_user_goal_once → run_llm_stream_with_retry → parse_canonical_tool_use / parse_xml_tool_use → execute_tool_call (loop_guard.check_call, tool.execute, record_outcome) → UIEventEmitter.emit_execution_event (thought, response, tool_call, tool_result, status, approval_request, review_pending) → en fin run_task retourne SGResult(ok, data).
3. **Sidebar** : append_text reçoit les événements (channel, content, phase, meta) → render_* (user, response, thinking, action, observation, terminal card, diff review, approval) → scroll_bottom, clear_draft_response si besoin.

## 6. Audit critique — points de vigilance

- **loop.py** : très gros fichier (~1400+ lignes) ; beaucoup de helpers _* pour la logique déterministe (list/read/write/run/repair, exact-content, strict target). À garder en tête pour refacto : regrouper par thème (path normalization, strict completion, repair, UI draft) si tu veux alléger.
- **paths.py** : seule source de vérité pour resolve_repo_path / read_repo_file ; pas de duplication côté outils — OK.
- **policy.py** : LoopGuard et RetryEngine bien séparés ; is_blocked centralise les commandes dangereuses — OK.
- **session.py** : TurnState et repair_conversation_history utilisés par loop ; DB SQLite simple — OK.
- **Orchestrator vs loop** : orchestrator = classification du tour (tool calls, final response) ; loop = exécution, TurnState, chemins déterministes. Frontière claire.
- **Parser** : parse_canonical_tool_use prioritaire, parse_xml_tool_use en fallback ; regex nombreuses mais regroupées — maintenable.
- **Lua** : sidebar.lua très long (rendu de tous les types d’événements) ; init.lua et rpc.lua restent lisibles. Pas de doublon évident avec le backend.
- **Pas de code mort repéré** dans les fichiers parcourus ; imports cohérents.
